# Project Rules

以下规则适用于当前仓库的整个目录树，作为默认工作规则自动生效。

## Scope

- 只修改与用户当前任务直接相关的文件、逻辑和配置。
- 不要为了“顺手修一下”去改无关模块、无关 warning、无关样式或无关脚本。
- 如果发现了与当前任务无关的问题，先明确说明，再等待用户确认是否一并处理。

## Delivery

- 每次改动都要以“可交付”为目标，优先提供最小且完整的改动闭环。
- 不要留下半完成状态的主流程改动；涉及流程功能时，应保证主链路可读、可验证、可交付。
- 最终说明中应清楚交代改动范围、关键文件、验证情况，以及未处理项是否因超出本次任务范围而刻意保留。
- 对关键流程或复杂逻辑补充必要注释，确保同事能够快速理解。

## Change Discipline

- 优先做根因修复，不要用会扩大改动面的绕行方案替代。
- 若某个报错或 warning 不属于当前任务主线，不要默认修复。
- 如果为了排查问题必须查看或验证其他模块，可以读取，但不要默认改动它们。

## 重构一致性检查

- 重构必须保持与旧代码的行为一致，不能只做视觉样式一致。
- 改动前需梳理并核对完整逻辑链路：e数据来源 -> 计算过程 -> 状态分支 -> UI 展示与交互。
- 状态枚举与分支必须完整保留（例如：`in/out/''`、`enabled/disabled`、`online/offline`），不得合并、删减或默认化处理。
- 若旧实现包含动画、轮询、节流、防抖、错误兜底等运行时行为，重构后必须保留等价行为。
- 未通过状态矩阵检查前不得标记完成；至少覆盖：正常态、边界值、零值/空值、异常态、权限/开关差异。

## Comment Preservation

- 除非用户明确要求，否则不要删除、改写或“顺手清理”现有注释。
- 进行代码改动时，默认保留改动点附近注释及其原本语义。
- 如因正确性必须调整注释，只允许做最小必要改动，并在最终说明中明确标注。
